03 / CRITICAL FLOWS

Follow the state transition.

The important architecture is not a folder tree—it is where authority moves between browser keys, app records, chain transactions, and indexed overlay state.

Economic core

Lock · like · mint

One client-built transaction atomically selects a live minter branch, locks sats, records social intent, mints a reward, and advances one or two successor branches.

Validate intent and unlocked keys

Post/LockSheet collects the target post; lockActions.ts requires owner and payment WIFs from session-scoped wallet state.

Select a spendable minter branch

GET /api/overlay/minters queries unspent BSV-21 events, filters the configured origin, excludes locally reserved outpoints, decodes candidates, and prefers remaining supply.

Build the contract transaction

The browser reconstructs LockLikeMintBSV21Parallel, derives payment and owner PKHs, fetches chain height, and calls buildTxForMint.

Fund, sign, and assemble BEEF

Payment UTXOs and parent transactions are prefetched. The wallet adds funding inputs/change while parent-complete BEEF ancestry is assembled for overlay validation.

Submit and index

POST /api/beef-cache builds the BEEF, submits the tagged transaction to the BSV21 overlay, and coordinates authenticated like persistence.

Mirror for application UX

Only after chain submission, /api/likes/submit writes the lock-like metadata to Supabase. Realtime updates then patch NEW feed caches and invalidate TOP ranking.

Subsidized publishing

Post / reply

The user proves authorship with the owner key; the application pays the small transaction fee from a server-held payment key.

Compose in the browser

Post or reply text and optional image payload are normalized into BSocial outputs.

Sign authorship locally

aip-signer.ts uses the unlocked owner WIF to produce the AIP signature and signer address. Raw owner keys never need to reach the server.

Fund on the server

POST /api/sign-and-pay validates the payload, appends AIP, gets WOC UTXOs for APP_PAYMENT_KEY, builds and signs the funded transaction.

Broadcast and persist

The transaction is broadcast, then /api/posts or /api/replies writes the authenticated user’s row into Supabase.

Distribute through Realtime

useRealtimeUpdates subscribes to Postgres INSERT events and React Query updates visible feeds and comment state.

Two separate gates

Login / wallet

A Supabase session authorizes app data. A separately unlocked local wallet authorizes signatures and spends.

Authenticate the account

Supabase email/password or Google OAuth establishes cookie-backed identity. proxy.ts refreshes sessions and protects wallet, profile, and vault pages.

Create two keys locally

The wallet generates an owner key for authorship/token ownership and a payment key for funding and lock spending.

Encrypt before persistence

Both WIFs are BIP38-encrypted in the browser with the same password. Supabase profiles store encrypted blobs and public addresses.

Unlock into session scope

The browser decrypts BIP38 and writes active WIFs/addresses to sessionStorage. Closing the session or logout removes the hot-wallet state.

Optional passkey convenience

WebAuthn helpers protect locally encrypted login or wallet secrets. They are device conveniences—not server-side passwordless identity and not a replacement for BIP38.

Chain first, mirror second

Vault unlock

Unlockability comes from the chain’s block height and output script; the Supabase likes table makes the workflow queryable.

Load mirrored locks

GET /api/likes partitions active, unlockable, and spent rows using current block height and unlock_height.

Resolve source transactions

unlockCoins.ts batches WOC transaction detail and spent-output checks, rejecting missing, unconfirmed, or already-spent inputs.

Build and sign locally

The browser spends lock output vout 2 with the payment key, sets locktime to the maximum required height, and calculates a size-based fee.

Broadcast

The raw unlock transaction is submitted through the broadcast gateway.

Reconcile the mirror

Likes are patched with is_spent and spent_txid; reconciliation routes repair divergence if broadcast succeeds but the database update fails.

Ordinal marketplace

Token trade

The configured origin ID binds market views and wallet actions to the same $VZN BSV-21 lineage.

Resolve token context

/api/vzn/token-info, /api/vzn/utxos, and overlay balance routes expose metadata, holdings, and live state for the configured origin.

Read market state

Trade tabs combine sale history, active ordinal listings, wallet token UTXOs, and current balance.

Construct the chosen action

WalletProvider lazy-loads js-1sat-ord and builds list, purchase, or cancel transactions with browser-held keys.

Broadcast and refresh

The resulting transaction is broadcast and client queries are refreshed to reflect balances, listings, sales, and “My Orders.”

04 / CONTRACT ANATOMY

The transaction is the rulebook.

LockLikeMintBSV21Parallel uses SIGHASH_ANYONECANPAY_ALL and commits to the exact output hash. Its mutable state is remaining branch supply and last height.

VOUT 0Successor Aceil(remaining / 2), or all remaining supply below split threshold
VOUT 1Successor BOptional second live minter branch when split threshold is met
VOUT 2*Locked satsPayment-key PKH, spendable after current height + blocks
NEXTMAP likeZero-sat OP_FALSE OP_RETURN: app · type=like · target txid
NEXT$VZN rewardBSV-21 transfer to owner-key PKH, capped by branch supply
NEXTService feeFixed output embedded in contract construction
FINALBSV changeWallet funding remainder after outputs and miner fee

* The lock is vout 2 for the common two-successor shape and vout 1 when only one successor remains. The current vault constant targets vout 2.

08 / TRUST BOUNDARIES

Keys, cookies, and irreversible state.

The architecture deliberately separates account identity, wallet authority, server subsidy, app indexing, and consensus-enforced value.

Browser trust zone

User wallet authority

  • Owner and payment private keys are generated and decrypted client-side.
  • Raw WIFs live in sessionStorage only while unlocked.
  • BIP38-encrypted WIFs are durable profile fields.
  • Lost BIP38 password means no recovery.
Application trust zone

Identity and subsidy

  • Supabase cookies identify the app user.
  • APP_PAYMENT_KEY subsidizes signed posts/replies.
  • SUPABASE_SERVICE_ROLE_KEY and ARC_API_KEY are server-only.
  • API routes validate payloads and mediate external services.
Consensus trust zone

Economic truth

  • Contract code enforces reward, lock, fee, and successor outputs.
  • Blockchain determines UTXO existence and spendability.
  • Overlay is an index, not consensus.
  • Supabase may temporarily diverge and must be reconciled.
